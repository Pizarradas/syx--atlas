05.3 — Contextos extremos (SYX Atlas)
Naturaleza de este documento

Este documento define cómo el sistema debe comportarse en condiciones donde:

el tiempo es mínimo
la atención es casi inexistente
el espacio es extremadamente limitado
la interacción es difícil o imprecisa
Principio base

Si el sistema funciona en condiciones extremas,
funcionará en cualquier otra situación.

1. Qué es un contexto extremo

Un contexto donde se reduce al mínimo:

atención
tiempo
precisión
capacidad de interacción
2. Tipos de contextos extremos
Movimiento (ej. coche)
atención fragmentada
interacción mínima
tiempo limitado
Micro-dispositivos (ej. smartwatch)
espacio extremadamente reducido
interacción muy limitada
Consumo rápido
el usuario no se detiene
escaneo inmediato
Condiciones adversas
mala iluminación
ruido
distracciones
3. Reducción al núcleo

El sistema debe reducirse a:

lo esencial
lo accionable
lo comprensible en segundos
4. Eliminación de lo no crítico

En contexto extremo, eliminar:

elementos decorativos
información secundaria
redundancias
5. Jerarquía radical

Debe ser evidente sin esfuerzo:

qué es lo más importante
qué puede ignorarse
6. Interacción mínima

El sistema debe requerir:

→ el menor número de acciones posible

7. Tiempo de comprensión

El contenido debe entenderse en:

→ segundos

8. Claridad absoluta

No puede haber:

ambigüedad
ruido
duda
9. Simplificación estructural

El layout debe:

reducirse
reorganizarse
priorizar
10. Feedback inmediato

Toda acción debe tener respuesta clara e instantánea.

11. Tolerancia a error

El sistema debe:

ser permisivo
evitar errores críticos
permitir recuperación rápida
12. Consistencia extrema

En condiciones difíciles, la consistencia es clave.

13. Eliminación de complejidad visual

Menos elementos:

→ mejor comprensión

14. Sistema de prioridad absoluta

El sistema debe decidir:

qué mostrar
qué ocultar
15. Adaptación sin ruptura

El sistema no crea una versión nueva.

Reduce el sistema existente.

16. Continuidad conceptual

Aunque simplifique:

mantiene lógica
mantiene reglas
mantiene coherencia
17. Evaluación en condiciones reales

El sistema debe probarse en:

uso en movimiento
uso rápido
interacción limitada
18. Señales de fallo
dificultad de comprensión
necesidad de detenerse
errores frecuentes
sobrecarga visual
19. Señales de éxito
comprensión inmediata
uso sin esfuerzo
claridad total
rapidez
20. Regla final

Si el sistema necesita atención plena para funcionar:

→ no está bien diseñado

Ejemplo simplificado

Contexto:

usuario en movimiento

↓

Sistema:

muestra solo titular
elimina secundarios
simplifica interacción

↓

Resultado:

uso rápido y efectivo

Cierre

El sistema no se demuestra en condiciones ideales.

Se demuestra cuando todo falla.
