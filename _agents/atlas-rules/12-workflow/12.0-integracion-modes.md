# 12.0 — Integración con SYX Modes (SYX Atlas)

## Naturaleza de este documento

Este documento define cómo Atlas usa los agentes técnicos de SYX (modes) para ejecutar tareas.

Los modes son herramientas agnósticas. No conocen Atlas.
Atlas es el modelo mental que decide qué pedir a cada mode, cómo formularlo y en qué orden.

---

## Principio base

> Atlas decide.
> Los modes ejecutan.
> Nunca al revés.

---

## 1. Modelo de relación

```
Atlas Rules (modelo mental de proyecto)
│
│  ¿Qué nivel editorial? ¿Qué jerarquía? ¿Qué proporción?
│  ¿Qué contexto de uso? ¿Qué densidad? ¿Qué tokens necesita?
│
├──► [SYX: UX]     → estructura HTML, estados, accesibilidad, comportamiento
├──► [SYX: TOKEN]  → tokens necesarios para las decisiones de Atlas
├──► [SYX: UI]     → implementación SCSS de lo que Atlas + UX han decidido
├──► [SYX: AUDIT]  → verificación técnica + verificación de contratos Atlas
├──► [SYX: SKETCH] → boceto rápido de una idea editorial
├──► [SYX: THEME]  → variante visual para una marca específica
└──► [SYX: MIGRATE]→ migración de legacy vars en páginas editoriales
```

Atlas no invoca los modes directamente con un prefijo combinado.
Atlas **enriquece la petición** antes de invocar el mode correcto.

---

## 2. Cómo formular una petición a los modes desde Atlas

### Estructura de una petición enriquecida

```
[SYX: MODE]: [descripción técnica de la tarea]

Contexto Atlas:
- Nivel editorial: N[1/2/3/4]
- Zona: [hero / grid editorial / secundarias / análisis]
- Jerarquía: [qué es N1, cuántos N2, etc.]
- Restricciones: [context-first, densidad, publicidad integrada, etc.]
- Tokens Atlas relevantes: [referencias a 08.0 si aplica]
```

El contexto Atlas le da al mode toda la información que necesita para tomar decisiones correctas sin conocer Atlas.

### Ejemplo — petición a UX mode

```
[SYX: UX]: Design the main editorial grid for an economy section.
           Two-column layout: news articles + market data sidebar.

Contexto Atlas:
- Nivel editorial: N2 (3 noticias destacadas, con imagen)
- Zona: org-grid-editorial — proporción 2:1 (contenido/sidebar)
- En mobile: una columna, sidebar después del contenido
- El sidebar contiene datos de mercado (bolsas, materias primas) + un bloque de publicidad 300×250
- Publicidad: integrada en el grid, nunca flotante (atlas-rules 06.3)
- Tipografía N2: atom-headline--lg con clamp(), según atlas-rules 10.0
```

### Ejemplo — petición a TOKEN mode

```
[SYX: TOKEN]: Define tokens for a 4-level editorial hierarchy card system.

Contexto Atlas:
- 4 niveles: N1 (hero), N2 (destacada), N3 (secundaria), N4 (complementaria)
- Escala tipográfica por nivel según atlas-rules/10-puentes/10.0-jerarquia-tipografia.md:
  N1: clamp(--type-xl, 4vw, --type-3xl)
  N2: clamp(--type-md, 2vw, --type-lg)
  N3: --type-base (fijo)
  N4: --type-sm (fijo)
- Spacing por nivel:
  N1: --semantic-space-xl alrededor
  N2: --semantic-space-lg entre cards
  N3: --semantic-space-md entre items
- Colores: usar la tabla de equivalencias en atlas-rules/12-workflow/12.1-bridge-tokens.md
```

### Ejemplo — petición a AUDIT mode

```
[SYX: AUDIT]: Review portada-economia.html

Scope: full audit — R01–R08 estándar más las siguientes verificaciones adicionales
requeridas por el proyecto Atlas:

Verificaciones adicionales:
1. @layer declarado con stack correcto
2. grid-template-columns sin px fijos en columnas editoriales
3. nth-child con índices concretos para gestión de bordes (anti-patrón)
4. Toggle de navegación mobile con JS implementado
5. Publicidad integrada en flujo normal (no position:absolute)
6. Un único elemento con atom-headline--xl/3xl (N1 único)
7. Sidebar de datos accesible en mobile (no oculto, reordenado)

Referencia de contratos: atlas-rules/09-contratos/09.0-contratos-y-validacion.md
```

## 3. Tabla de asignación: tarea Atlas → mode SYX

| Tarea Atlas | Mode principal | Mode de apoyo |
|---|---|---|
| Diseñar una zona editorial (hero, grid, secundarias) | UX | — |
| Especificar comportamiento de un nivel editorial | UX | — |
| Definir tokens para jerarquía editorial | TOKEN | — |
| Implementar SCSS de un componente editorial | UI | TOKEN (si faltan tokens) |
| Crear variante visual para una marca | THEME | TOKEN |
| Auditar una página editorial | AUDIT | — |
| Hacer boceto de una idea editorial | SKETCH | — |
| Explorar técnicas visuales avanzadas para editorial | CREATIVE | — |
| Migrar variables legacy en páginas editoriales | MIGRATE | — |
| Secuencia completa: diseño → tokens → implementación → audit | UX → TOKEN → UI → AUDIT | — |

## 4. Qué Atlas aporta a cada mode

| Mode | Lo que Atlas aporta antes de invocar |
|---|---|
| UX | Nivel editorial (N1/N2/N3/N4), zona, densidad esperada, restricciones context-first, comportamiento mobile, publicidad si aplica |
| TOKEN | Escala tipográfica por nivel (10.0), equivalencias de nombres (12.1), tokens primitivos ya definidos en 08.0 |
| UI | Decisiones UX ya tomadas, nivel editorial, checklist mobile-first específico (12.2), proporciones de layout |
| AUDIT | Contratos adicionales del proyecto (09.0), checks mobile-first del proyecto (12.2), expectativas de jerarquía editorial |
| SKETCH | Orden de zonas editoriales por prioridad, restricciones de colapso mobile, nivel de cada zona |
| THEME | Paleta de la marca, relación con los tokens de Atlas (12.1) |
| MIGRATE | Variable legacy, su equivalente en el sistema Atlas (12.1) |

---

## 5. Lo que los modes NO necesitan saber de Atlas

Los modes son agnósticos. No necesitan:
- Conocer los principios fundacionales de Atlas (00–03)
- Conocer el marco matemático de Atlas (04)
- Conocer la lógica editorial de Atlas (06) en detalle
- Conocer el sistema de gobierno de Atlas (11)

Solo necesitan recibir las **decisiones ya tomadas** por Atlas, expresadas en el lenguaje del mode.
