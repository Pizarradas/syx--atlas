# 12.3 — Flujos editoriales: secuencias completas de workflow (SYX Atlas)

## Naturaleza de este documento

Este documento define las secuencias de trabajo completas para las tareas editoriales más frecuentes en SYX Atlas.

Cada flujo especifica: qué se hace, en qué orden, qué mode ejecuta cada paso, qué artefactos produce y cuándo está terminado.

---

## Principio base

> Un flujo no es una secuencia de pasos.  
> Es una cadena de responsabilidades.  
> Cada paso recibe una decisión y entrega otra.

---

## 1. Flujo A — Diseño completo de portada editorial

**Cuándo**: crear una portada editorial nueva (portada general, sección, portada de marca).  
**Duración típica**: 4–6 turnos.  
**Resultado**: página HTML standalone auditada y lista para promoción a SCSS.

### Secuencia

```
Paso 1 — Atlas: definir la jerarquía editorial
  Responsable: Atlas Rules (manual, antes de invocar ningún mode)
  Input: brief del proyecto
  Output: estructura N1/N2/N3/N4 con densidad y zonas
  Verificar: un solo N1, 3–4 N2 máximo, densidad coherente con contexto

Paso 2 — [SYX: SKETCH]: boceto de la estructura
  Responsable: SKETCH mode
  Input: jerarquía definida en Paso 1 + contexto Atlas (06.1, 06.2)
  Output: HTML único con layout básico, valores hardcoded permitidos
  Verificar: columna única en mobile, orden DOM correcto, handoff comment

Paso 3 — [ATLAS + SYX: UX]: estructura HTML completa con estados
  Responsable: UX mode con contexto Atlas
  Input: boceto del Paso 2 + jerarquía del Paso 1
  Context Atlas obligatorio:
    - Niveles: N1/N2/N3 + densidad esperada
    - Zonas: hero / grid editorial / sidebar / publicidad
    - Comportamiento mobile: qué se colapsa, qué se reordena
    - Publicidad: dónde se integra (06.3)
  Output: estructura HTML semántica con todos los estados y ARIA

Paso 4 — [SYX: TOKEN]: verificar y completar tokens
  Responsable: TOKEN mode
  Input: HTML del Paso 3 + tokens.json
  Context Atlas obligatorio:
    - Tabla de equivalencias de 12.1
    - Escala tipográfica por nivel de 10.0
    - Formato OKLCH para primitivos si van a SCSS
  Output: tokens nuevos definidos o confirmación de que existen

Paso 5 — [ATLAS + SYX: AUDIT]: auditoría combinada
  Responsable: AUDIT mode con checks Atlas (governance/05-audit.md)
  Input: HTML + CSS del flujo completo
  Output: reporte con Capa 1 (R01–R08) + Capa 2 (contratos Atlas) + context-first checks (12.2)
  Condición de aprobación: PASS o PASS WITH WARNINGS en ambas capas
```

### Artefactos finales

| Artefacto | Ubicación |
|---|---|
| Página HTML auditada | `portada-[nombre].html` en raíz o subcarpeta de proyecto |
| Tokens nuevos documentados | `mind-system/atlas-rules/08-tokens/08.0-registro-tokens.md` actualizado |
| Reporte de auditoría | `AUDIT_REPORT.md` o comentario en el HTML |

## 2. Flujo B — Implementación SCSS de componente editorial

**Cuándo**: promover un componente editorial de HTML standalone a SCSS del sistema.  
**Duración típica**: 3–4 turnos.  
**Resultado**: componente SCSS con tokens, registrado en `component-registry.json`, contrato validado.

### Secuencia

```
Paso 1 — Verificación previa
  Responsable: manual (antes de invocar modes)
  Verificar:
    □ ¿El componente existe ya en component-registry.json? Si sí → usar el existente.
    □ ¿Los tokens que necesita existen en tokens.json? Si no → ir a Flujo C primero.
    □ ¿El nivel editorial está claro (atom / mol / org)?

Paso 2 — [SYX: TOKEN]: verificar tokens del componente
  Input: lista de propiedades del componente (color, tipo, spacing, layout)
  Context Atlas: tabla 12.1 para mapear conceptos Atlas a nombres SYX
  Output: confirmación de tokens existentes + lista de tokens a crear si faltan

Paso 3 — [ATLAS + SYX: UI]: implementar SCSS
  Input: HTML de referencia + tokens verificados en Paso 2
  Context Atlas obligatorio:
    - Nivel editorial del componente (N1/N2/N3/N4)
    - Pre-flight adicional de 12.2 (grid proporcional, nth-child, etc.)
    - @layer stack de Modes (no el stack de Atlas 09.0)
  Output: archivo SCSS del componente + actualización de component-registry.json

Paso 4 — Validación
  Comando: node scripts/syx-validate.js
  Verificar: R01–R04 sin violaciones
  Si hay violaciones → corregir en el mismo turno antes de entregar
```

### Artefactos finales

| Artefacto | Ubicación |
|---|---|
| Archivo SCSS | `scss/[atoms|molecules|organisms]/[nombre]/[nombre].scss` |
| Registro actualizado | `component-registry.json` |
| Tokens nuevos | `scss/abstracts/tokens/` + `tokens.json` actualizado |
| Validación limpia | Output de `syx-validate.js` sin errores |

## 3. Flujo C — Definición de tokens para sistema editorial nuevo

**Cuándo**: el sistema editorial necesita tokens que aún no existen.  
**Duración típica**: 2–3 turnos.  
**Resultado**: tokens definidos en la cadena Primitive → Semantic → Component, registrados.

### Secuencia

```
Paso 1 — Atlas: mapear necesidades conceptuales
  Responsable: Atlas Rules (manual)
  Input: qué conceptos editoriales necesitan tokens (colores, tipo, spacing, jerarquía)
  Output: lista de necesidades en lenguaje Atlas
    Ejemplo: "color de categoría economía", "titular N1 de portada económica"

Paso 2 — Traducción mediante bridge
  Consultar 12.1 para convertir lenguaje Atlas a naming SYX.
  Si el concepto ya tiene equivalente en 12.1 → usar el token existente.
  Si no tiene equivalente → definir nuevo token (Paso 3).

Paso 3 — [SYX: TOKEN]: crear los tokens que faltan
  Input: lista de tokens a crear + jerarquía Primitive → Semantic → Component
  Context Atlas obligatorio:
    - Formato OKLCH para primitivos de color (si van a SCSS)
    - Escala tipográfica de 10.0 para tokens de tipo
    - No duplicar tokens existentes en tokens.json
  Output:
    - Primitivos nuevos (si aplica): `--primitive-color-[nombre]` en OKLCH
    - Semánticos nuevos: `--semantic-color-[función]` mapeados desde primitivos
    - Componente nuevos (si son específicos): `--component-[nombre]-[propiedad]`

Paso 4 — Registro y documentación
  Actualizar:
    - tokens.json con los nuevos tokens
    - mind-system/atlas-rules/08-tokens/08.0-registro-tokens.md con la cadena completa
    - 12.1-bridge-tokens.md si hay un nuevo concepto Atlas con equivalente SYX
```

### Artefactos finales

| Artefacto | Ubicación |
|---|---|
| Tokens SCSS | `scss/abstracts/tokens/` |
| Registro técnico | `tokens.json` |
| Registro conceptual | `mind-system/atlas-rules/08-tokens/08.0-registro-tokens.md` |
| Bridge actualizado (si aplica) | `12.1-bridge-tokens.md` |

## 4. Flujo D — Auditoría editorial de página existente

**Cuándo**: revisar una página editorial ya construida (portada, sección, layout de marca).  
**Duración típica**: 1–2 turnos.  
**Resultado**: reporte de violaciones categorizado, con fixes concretos.

### Secuencia

```
Paso 1 — Contexto previo a la auditoría
  Antes de invocar AUDIT mode, determinar:
    □ ¿Es una página editorial standalone (HTML + CSS inline)?
    □ ¿Es una página compilada desde SCSS del sistema?
  Esto define qué contratos se aplican (ver governance/05-audit.md).

Paso 2 — [ATLAS + SYX: AUDIT]: auditoría combinada
  Scope: Capa 1 (R01–R08) + Capa 2 (contratos Atlas) + checks context-first (12.2)
  Input: archivo a auditar + contexto de si es standalone o compilado
  Formato de petición a AUDIT:
    "[ATLAS + SYX: AUDIT]: Auditar [archivo]
     Tipo: [standalone HTML / SCSS compilado]
     Nivel editorial esperado: [descripción breve — ej: portada de economía]
     Contratos adicionales: ver mind-system/atlas-rules/09-contratos/09.0-contratos-y-validacion.md"

Paso 3 — Priorización de fixes
  AUDIT entrega el reporte. Atlas prioriza:
    1. Violaciones que afectan accesibilidad (CA-*) → siempre primero
    2. Violaciones de jerarquía editorial (10.0) → N1 único, contraste entre niveles
    3. Violaciones de layout (CL-*) → grid sin px, nth-child
    4. Violaciones de contratos de componentes (CC-*)
    5. Violaciones técnicas R01–R08 → si el archivo es SCSS compilado

Paso 4 — Fix (si se solicita en el mismo flujo)
  Para fixes en HTML standalone → no requiere TOKEN o UI mode. Corregir directo.
  Para fixes en SCSS → seguir Flujo B (implementación SCSS).
```

### Formato del reporte (referencia)

```
## Capa 1 — Contratos Modes (R01–R08)
[solo si el archivo tiene SCSS compilado; omitir en HTML standalone]

## Capa 2 — Contratos Atlas
| Contrato | Línea | Violación | Fix |
|---|---|---|---|

## Context-first checks (12.2)
| Check | Resultado | Línea | Nota |
|---|---|---|---|

## Jerarquía editorial detectada
N1: [elemento]
N2: [elementos — debe ser ≤ 4]
N3: [elementos]
Problemas: [si los hay]

## Verdict
PASS / FAIL / PASS WITH WARNINGS
```

## 5. Flujo E — Creación de variante de tema para portada editorial

**Cuándo**: adaptar una portada editorial existente a una marca específica con su paleta y tipografía.  
**Duración típica**: 2–3 turnos.  
**Resultado**: tema SCSS con tokens de marca integrados con la jerarquía editorial de Atlas.

### Secuencia

```
Paso 1 — Atlas: identificar las decisiones editoriales de la marca
  Definir antes de invocar modes:
    - ¿La marca tiene variación en la jerarquía editorial (distinto peso N1)?
    - ¿Los colores de categoría son distintos (deporte, economía, cultura)?
    - ¿La tipografía de la marca cambia los ratios de contraste entre niveles (10.0)?

Paso 2 — [SYX: TOKEN]: verificar tokens de marca en bridge
  Consultar 12.1 para mapear colores de marca a tokens semánticos SYX.
  Verificar: ¿el --semantic-color-primary de la marca mantiene ≥ 4.5:1 sobre el fondo?

Paso 3 — [SYX: THEME]: crear el tema
  Input: paleta de la marca + decisiones editoriales del Paso 1
  Context Atlas obligatorio:
    - Los tokens semánticos deben respetar los ratios de contraste tipográfico de 10.0
    - Si la tipografía de la marca altera el ratio N1/N2 → señalarlo explícitamente
  Output: tema SCSS con los 4 bundle contexts (light, dark, print, high-contrast)

Paso 4 — [ATLAS + SYX: AUDIT]: verificar que el tema no rompe la jerarquía editorial
  Checks específicos:
    - ¿El color del titular N1 mantiene contraste ≥ 4.5:1 sobre el fondo del tema?
    - ¿Los colores de categoría son distinguibles entre sí (no confundibles)?
    - ¿El tema funciona en modo de alto contraste del OS?
```

## 6. Decisión rápida: ¿qué flujo uso?

| Situación | Flujo |
|---|---|
| Construir una portada desde cero | A — Diseño completo de portada |
| Pasar un componente HTML a SCSS | B — Implementación SCSS |
| Faltan tokens para un nuevo componente | C — Definición de tokens |
| Revisar una página ya construida | D — Auditoría editorial |
| Adaptar una portada a una marca | E — Variante de tema |
| Solo boceto rápido para validar idea | SKETCH directo, sin flujo completo |
| Solo verificar tokens existentes | TOKEN directo, sin flujo completo |

---

## 7. Reglas transversales a todos los flujos

1. **No saltar pasos**: cada paso recibe la decisión del anterior. No pasar a UI sin TOKEN verificado.
2. **El reporte de AUDIT es la condición de cierre**: ningún flujo que incluya construcción está terminado sin auditoría.
3. **Las decisiones Atlas se toman fuera de los modes**: los modes ejecutan, no deciden. La jerarquía N1/N2/N3, la densidad y las zonas se definen antes de invocar cualquier mode.
4. **Los artefactos se registran**: tokens nuevos → `tokens.json`. Componentes nuevos → `component-registry.json`. Excepciones → `11.0-arbitraje-principios.md`.
5. **Context-first es transversal**: cualquier artefacto producido en cualquier flujo pasa los checks de `12.2` antes de darse por terminado.

---

## Cierre

Un flujo define cuándo está terminado algo.

Sin eso, cada paso es un inicio que no acaba.
