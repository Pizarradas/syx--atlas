# 11.1 — Versionado de reglas (SYX Atlas)

## Naturaleza de este documento

Este documento define cómo evoluciona el sistema de reglas sin perder coherencia ni trazabilidad.

Un sistema que no puede cambiar es frágil.
Un sistema que cambia sin control es caótico.

---

## Principio base

> Las reglas pueden mejorar.  
> Pero el proceso de mejora es parte del sistema.

---

## 1. Versión actual del sistema de reglas

**Versión**: 1.0  
**Fecha**: 2025  
**Estado**: activo

---

## 2. Tipos de cambio

Los cambios en las reglas se clasifican en tres tipos:

| Tipo | Descripción | Ejemplos |
|---|---|---|
| **Corrección** | Arregla una incoherencia sin cambiar la lógica del sistema | Renombrar un token mal nombrado, corregir un ratio de contraste erróneo |
| **Extensión** | Añade una regla nueva sin modificar las existentes | Nuevo tipo de componente, nueva zona editorial, nueva categoría de color |
| **Revisión** | Modifica una regla existente, cambia su comportamiento o elimina una regla | Cambiar la jerarquía de escalas, redefinir un nivel editorial |

---

## 3. Quién puede proponer cada tipo

| Tipo | Puede proponer | Requiere aprobación de |
|---|---|---|
| Corrección | Cualquier implementador, Claude | El documento queda registrado; no bloquea implementación |
| Extensión | Cualquier implementador, Claude | Revisión antes de que otras implementaciones la adopten |
| Revisión | Solo el equipo de diseño del sistema | Revisión formal; afecta implementaciones existentes |

---

## 4. Proceso de cambio

### Para Correcciones

```
1. Identificar el documento afectado
2. Describir la incoherencia (qué dice, qué debería decir)
3. Aplicar el cambio directamente en el documento
4. Registrar en el changelog de este documento (sección 8)
5. Verificar que no hay implementaciones que dependan del valor incorrecto
```

### Para Extensiones

```
1. Definir la nueva regla con el formato del documento correspondiente
2. Verificar que no contradice ninguna regla existente (consultar 11.0)
3. Añadir el archivo o la sección al lugar correcto de la estructura
4. Registrar en changelog (sección 8)
5. Actualizar referencias en documentos que la deban citar
```

### Para Revisiones

```
1. Documentar el problema que genera la regla actual
2. Proponer la nueva formulación
3. Identificar todas las implementaciones afectadas
4. Revisar y actualizar implementaciones antes de publicar el cambio
5. Marcar la regla anterior como deprecada durante un período de transición
6. Registrar en changelog con tipo BREAKING si afecta comportamiento externo
```

---

## 5. Cómo deprecar una regla

Una regla deprecada no se elimina inmediatamente. Se marca y se da un período de transición.

**Proceso**:
1. Añadir al inicio del documento o sección afectada:

```
> ⚠ DEPRECADO en v[X.Y] — [fecha]  
> Motivo: [razón]  
> Reemplazado por: [referencia al nuevo documento/sección]  
> Eliminación prevista: [fecha o versión]
```

2. Registrar en el changelog como `DEPRECATED`.
3. Actualizar las implementaciones existentes durante el período de transición.
4. Eliminar el bloque deprecado en la versión indicada.

---

## 6. Cómo Claude debe tratar las reglas

Cuando Claude trabaja con este sistema de reglas:

1. **Verificar fecha y estado** de los documentos que consulta. Una regla marcada como deprecada no debe aplicarse.
2. **Detectar incoherencias** entre reglas durante la implementación → proponer corrección siguiendo el proceso de la sección 4.
3. **No inventar reglas**. Si falta una regla para una situación, señalarlo y escalar. No asumir.
4. **Citar la regla consultada** cuando toma una decisión. Esto permite auditar si la decisión es correcta o si la regla necesita revisión.
5. **Si una regla parece incorrecta**: no aplicarla silenciosamente ni ignorarla. Señalar la posible incoherencia antes de implementar.

---

## 7. Reglas de nomenclatura de versiones

El sistema usa versionado semántico adaptado a sistemas de diseño:

```
MAJOR.MINOR
  │     └── Extensiones: nuevas reglas, nuevos componentes, correcciones
  └──────── Revisiones: cambios que modifican comportamiento existente
```

- `v1.0` → sistema inicial
- `v1.1` → primera extensión (nuevos tokens, nuevos documentos)
- `v2.0` → revisión estructural que afecta implementaciones existentes

**No existe versión de parche**: las correcciones menores se registran pero no cambian el número de versión.
