# 09.0 — Contratos y validación (SYX Atlas)

## Naturaleza de este documento

Este documento define los contratos que toda implementación debe cumplir y cómo verificarlos.

Un contrato es una regla verificable.
Si no puede verificarse, no es un contrato; es una intención.

---

## Principio base

> Una implementación no está terminada cuando el código funciona.  
> Está terminada cuando cumple todos sus contratos.

---

## 1. Qué es un contrato

Un contrato define:

- **Qué** debe cumplir un componente, layout o token
- **Cómo** verificarlo
- **Qué constituye** una violación

Todo contrato es binario: se cumple o no se cumple.

---

## 2. Contratos de tokens

### CT-01 — Cadena Primitive → Semantic

**Contrato**: ningún componente o página usa un token primitivo directamente.

**Verificación**: buscar en CSS `--primitive-` fuera de `:root` o de la declaración de semánticos.  
Si aparece en un componente → violación.

---

### CT-02 — Sin valores crudos

**Contrato**: ningún valor de color, spacing o tipografía aparece fuera del sistema de tokens.

**Verificación**: buscar en CSS:
- Valores hex (`#[0-9a-fA-F]{3,6}`) fuera de `:root`
- `px` de spacing no derivados de tokens
- `font-size` con valores literales en componentes

Si aparecen → violación.

---

### CT-03 — Token declarado antes de usar

**Contrato**: todo token en uso existe en `08.0-registro-tokens.md`.

**Verificación**: comparar `var(--*)` del CSS implementado contra el inventario de `08.0`.  
Si hay tokens no registrados → violación.

---

## 3. Contratos de componentes

### CC-01 — Nomenclatura por nivel atómico

**Contrato**: los prefijos de clase respetan la capa arquitectónica.

| Capa | Prefijo |
|---|---|
| Atom | `.atom-` |
| Molecule | `.mol-` |
| Organism | `.org-` |

**Verificación**: ningún componente de capa molecule usa `.atom-` como clase raíz y viceversa.  
Si hay mezcla de prefijos en la misma capa → violación.

---

### CC-02 — Reutilización antes que creación

**Contrato**: antes de crear un componente nuevo, se ha comprobado si existe uno que pueda extenderse.

**Verificación**: revisión de inventario de componentes (este documento, sección 7).  
Si existe un componente similar sin modificador adecuado → el nuevo componente es violación por fragmentación.

---

### CC-03 — Componente no depende de contenido específico

**Contrato**: la apariencia de un componente no depende de una cantidad fija de elementos hijos.

**Verificación**: comprobar si el CSS usa `nth-child` con valores concretos para gestionar bordes o layout.  
Si la presentación se rompe con ±1 elemento → violación.

---

### CC-04 — Estados de interacción completos

**Contrato**: todo elemento interactivo declara los estados: default, hover, focus, active, disabled.

**Verificación**: buscar en CSS `cursor: pointer` o `role="button"` y verificar que existen estilos para `:hover`, `:focus-visible`, `:active`.  
Si falta algún estado → violación.

---

### CC-05 — Comportamiento funciona sin JS

**Contrato**: la estructura visible en carga inicial (sin JS ejecutado) es semánticamente correcta y no presenta elementos interactivos sin comportamiento definido.

**Verificación**: cargar la página con JS desactivado. Elementos con `onclick` o `cursor: pointer` sin JS deben seguir siendo estructuralmente coherentes.  
Si hay controles de navegación visibles e inoperables → violación.

---

## 4. Contratos de layout

### CL-01 — Sin anchos fijos en columnas proporcionales

**Contrato**: las columnas del grid se definen en `fr`, `%` o `clamp()`. No en `px` absolutos.

**Verificación**: buscar `grid-template-columns` con valores `px` concretos.  
Si aparecen → violación. Excepción permitida: imágenes con `aspect-ratio` definido dentro de un wrapper proporcional.

---

### CL-02 — Comportamiento coherente sin contenido

**Contrato**: el layout no se rompe cuando un bloque está vacío o tiene contenido mínimo.

**Verificación**: probar el layout con: 0 elementos, 1 elemento, el doble de elementos previstos.  
Si el layout se desestructura en cualquier caso → violación.

---

### CL-03 — Sin `@layer` ausente

**Contrato**: toda hoja de estilos de implementación declara `@layer` con el stack obligatorio.

**Stack obligatorio**:
```css
@layer reset, base, tokens, atoms, molecules, organisms, utilities;
```

**Verificación**: buscar en CSS si existe la declaración `@layer`.  
Si no existe → violación.

---

### CL-04 — Sin `!important`

**Contrato**: no se usa `!important` en ninguna declaración de implementación.

**Verificación**: buscar `!important` en CSS.  
Si aparece → violación. La especificidad se gestiona con `@layer` y arquitectura.

---

## 5. Contratos de accesibilidad

### CA-01 — Jerarquía de headings coherente

**Contrato**: la página tiene un único `<h1>`. Los headings no saltan niveles (no puede haber `<h3>` sin `<h2>` previo).

**Verificación**: mapear todos los `<hN>` del documento y verificar secuencia.

---

### CA-02 — Elementos interactivos tienen nombre accesible

**Contrato**: todo `<button>`, `<a>`, `<input>` tiene texto visible o `aria-label`.

**Verificación**: buscar elementos interactivos sin texto hijo ni `aria-label` ni `aria-labelledby`.  
Si existen → violación.

---

### CA-03 — Focus visible

**Contrato**: el sistema declara `:focus-visible` global con un estilo perceptible (no `outline: none` sin reemplazo).

**Verificación**: navegar la página con Tab. El foco debe ser visible en todo momento.  
Si el foco desaparece en algún elemento → violación.

---

### CA-04 — Contraste mínimo

**Contrato**: todas las parejas texto/fondo cumplen ratio ≥ 4.5:1 (AA).

**Verificación**: usar la tabla de contrastes de `08.1`. Cualquier pareja no registrada debe verificarse antes de implementar.

---

## 6. Contratos de publicidad

### CP-01 — Publicidad dentro del grid

**Contrato**: los bloques publicitarios ocupan posiciones previstas en el grid. No flotan ni interrumpen el flujo de lectura con `position: absolute/fixed`.

**Verificación**: inspeccionar la posición del bloque `.mol-publicidad` en el DOM y en el layout.  
Si está fuera del flujo normal → violación.

---

### CP-02 — Publicidad identificada

**Contrato**: todo bloque publicitario tiene una etiqueta visible que lo identifica como publicidad.

**Verificación**: buscar `.mol-publicidad` sin `.mol-publicidad__label` hijo.  
Si no hay etiqueta → violación.

## 8. Proceso de validación antes de entregar

```
1. CT-01: ¿algún componente usa primitivos directamente? → corregir
2. CT-02: ¿hay valores hex/px crudos en componentes? → corregir
3. CL-03: ¿existe @layer declarado? → añadir si falta
4. CL-04: ¿existe algún !important? → refactorizar
5. CC-03: ¿hay nth-child con números concretos para layout? → revisar
6. CC-05: ¿todos los controles interactivos tienen JS implementado? → completar
7. CA-01: ¿jerarquía de headings correcta? → verificar
8. CA-02: ¿todos los interactivos tienen nombre accesible? → verificar
9. CA-03: ¿focus visible en toda la página? → verificar tabulando
10. CP-02: ¿toda publicidad tiene etiqueta visible? → verificar
```

---

## Cierre

Los contratos no son opcionales.

Son el mecanismo mediante el cual el sistema se verifica a sí mismo.

Una implementación sin contratos verificados no es una implementación del sistema.
