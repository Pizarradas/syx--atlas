Naturaleza de este documento

Este documento define cómo evaluar el sistema de forma:

objetiva
medible
repetible

No se valida por opinión.
Se valida por comportamiento.

Principio base

Si no puede medirse, no puede mejorarse.

1. Qué se valida

El sistema debe validarse en tres niveles:

Nivel 1 — Estructural
coherencia del layout
consistencia de reglas
ausencia de excepciones
Nivel 2 — Cognitivo
facilidad de comprensión
rapidez de lectura
claridad jerárquica
Nivel 3 — Operativo
rendimiento
adaptabilidad
escalabilidad
2. Métricas estructurales
Consistencia

¿Se aplican las mismas reglas en todo el sistema?

Reutilización

¿Cuántas soluciones son reutilizables?

Complejidad

Número de:

reglas
variantes
excepciones
Coherencia

¿Las partes responden a una misma lógica?

3. Métricas cognitivas
Tiempo de comprensión

Tiempo que tarda el usuario en entender:

qué está viendo
qué puede hacer
Escaneabilidad

Capacidad de recorrer el contenido rápidamente.

Identificación de jerarquía

¿Se percibe claramente qué es importante?

Carga mental

Número de decisiones que debe tomar el usuario.

4. Métricas de densidad
Elementos por bloque

Cuántos elementos visibles hay.

Relación contenido / espacio

Equilibrio entre:

información
aire
Saturación

Detectar exceso de estímulos.

5. Métricas de ritmo
Regularidad

Consistencia del spacing.

Fluidez

Facilidad de recorrido vertical.

Rupturas

Número de interrupciones en el flujo.

6. Métricas de layout
Estabilidad

¿El layout se comporta de forma coherente en distintos contextos?

Continuidad

¿La transición entre tamaños es fluida?

Adaptabilidad

¿funciona en condiciones restrictivas?

7. Métricas de sistema
Número de reglas

Menos reglas → mejor sistema (si mantiene coherencia)

Número de excepciones

Cada excepción reduce calidad del sistema.

Escalabilidad

Capacidad de crecer sin romperse.

8. Validación cualitativa

No todo es numérico.

También se evalúa:

claridad percibida
sensación de orden
facilidad de uso
9. Pruebas reales

El sistema debe validarse en:

mobile real
condiciones imperfectas
uso rápido
uso distraído
10. Principio de falsabilidad

El sistema debe poder fallar.

Si no puede fallar, no puede mejorar.

11. Sistema de feedback

Los errores deben:

detectarse
analizarse
convertirse en reglas
12. Validación continua

El sistema no se valida una vez.

Se valida constantemente.

13. Relación con IA

Claude debe:

evaluar sus propias soluciones
detectar incoherencias
proponer mejoras
14. Señales de fallo del sistema
necesidad de excepciones
decisiones arbitrarias
incoherencias
dificultad de uso
15. Señales de sistema sano
coherencia
simplicidad
facilidad de uso
reutilización
16. Regla de mejora

Cada mejora debe:

simplificar
unificar
reforzar coherencia
17. Evitar validación subjetiva

No basarse en:

gustos
tendencias
opiniones aisladas
18. Trazabilidad de decisiones

Cada decisión debe poder rastrearse:

problema → regla → solución → resultado

19. Métrica clave

Menor esfuerzo cognitivo con mayor claridad estructural

20. Regla final

Si no se puede demostrar que algo mejora el sistema:

→ no debe incorporarse

Ejemplo simplificado

Problema:

Usuario tarda en entender la portada

↓

Medición:

tiempo de comprensión alto
demasiados elementos visibles

↓

Acción:

reducir densidad
reforzar jerarquía

↓

Resultado:

mejor comprensión

Cierre

Un sistema sin validación es una opinión organizada.

SYX Atlas debe ser un sistema verificable.

Lectura correcta de este documento

Este documento define cómo medir el sistema.
