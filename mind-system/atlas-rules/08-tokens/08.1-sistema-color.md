# 08.1 — Sistema de color (SYX Atlas)

## Naturaleza de este documento

Este documento define cómo funciona el color dentro del sistema.

El color no es un recurso estético.
Es información estructural.

---

## Principio base

> El color no decora.  
> El color comunica jerarquía, estado y significado.

---

## 1. Roles del color

El sistema define cinco roles para el color. Ningún color puede usarse fuera de su rol sin justificación estructural.

| Rol | Función | Tokens semánticos |
|---|---|---|
| **Superficie** | Fondos y planos de contenido | `--semantic-color-bg`, `--semantic-color-surface` |
| **Texto** | Legibilidad y jerarquía tipográfica | `--semantic-color-text-primary`, `--secondary`, `--muted` |
| **Borde** | Separación y estructura | `--semantic-color-border`, `--border-strong` |
| **Marca** | Identidad, acciones, interacción | `--semantic-color-brand`, `--brand-dark`, `--brand-hover` |
| **Estado** | Información funcional: positivo, negativo, neutro | `--semantic-color-alza`, `--baja`, `--neutra` |

---

## 2. El color en los niveles de jerarquía editorial

La jerarquía del contenido (definida en `06.1`) se traduce al color así:

| Nivel | Rol del color |
|---|---|
| N1 — Principal | Máximo contraste. Texto primario sobre superficie. Posible uso de marca como acento. |
| N2 — Destacado | Texto primario. Superficie diferenciada o borde de énfasis. |
| N3 — Secundario | Texto secundario. Superficie de página (no tarjeta). |
| N4 — Complementario | Texto muted. Sin superficie propia. |

**Regla**: la jerarquía de color debe reforzar la jerarquía editorial. Nunca contravenirla.

---

## 3. Contraste mínimo obligatorio

Todo texto debe cumplir WCAG 2.1 nivel AA como mínimo.

| Pareja | Ratio | Nivel |
|---|---|---|
| `text-primary` sobre `surface` | 16.8:1 | AAA |
| `text-secondary` sobre `surface` | 10.7:1 | AAA |
| `text-muted` sobre `surface` | 4.6:1 | AA |
| `brand` sobre `surface` | 7.2:1 | AA |
| `alza` sobre `surface` | 5.1:1 | AA |
| `baja` sobre `surface` | 5.8:1 | AA |
| Texto blanco sobre `brand` | 4.5:1 | AA |

**Regla**: antes de usar un color nuevo, verificar ratio de contraste.  
**Regla**: texto sobre imagen requiere overlay o fondo sólido. No texto directo sobre foto.

---

## 4. Color y estados de interacción

Todo elemento interactivo tiene una secuencia de color por estado. No puede haber estados sin color definido.

| Estado | Regla de color |
|---|---|
| Default | Color semántico estándar |
| Hover | Variante darker del token de marca (`--semantic-color-brand-hover`) o `surface-hover` para superficies |
| Focus | `--primitive-shadow-focus` (ring azul) — nunca solo color, siempre borde visible |
| Active | Mismo que hover con opacidad reducida |
| Disabled | `text-muted` sobre `bg`. Nunca usar colores de estado (alza/baja) en disabled |
| Error | `--semantic-color-baja` y `--semantic-color-baja-bg` |

---

## 5. Color en datos de mercado

Los indicadores financieros tienen un sistema de color funcional estricto.

| Situación | Color texto | Color fondo | Token |
|---|---|---|---|
| Valor positivo (alza) | `--semantic-color-alza` | `--semantic-color-alza-bg` | `atom-indicador--alza` |
| Valor negativo (baja) | `--semantic-color-baja` | `--semantic-color-baja-bg` | `atom-indicador--baja` |
| Sin variación | `--semantic-color-neutra` | ninguno | `atom-indicador--neutra` |

**Regla**: en ticker sobre fondo oscuro, los colores de alza/baja se ajustan para mantener contraste. Ver implementación de ticker.

---

## 6. Color de etiquetas de categoría

Las etiquetas de sección tienen color propio por categoría. El criterio de asignación es:

- Cada categoría tiene una familia de color diferenciada (azul, verde, naranja…)
- El color de texto debe tener ratio ≥ 4.5:1 sobre su fondo de etiqueta
- Los colores de etiqueta **no son primitivos sueltos**: deben registrarse como tokens de componente derivados de semánticos

| Categoría | Patrón de derivación |
|---|---|
| Economía | Familia `brand` (azul) |
| Mercados | Familia `success` (verde) |
| Empresas | Familia `warning` (naranja/marrón) |
| Banca | Familia `info` (azul claro) |
| Opinión | Familia `purple` — requiere añadir primitivo `--primitive-color-purple-*` |
| Tecnología | Familia `teal` — requiere añadir primitivo `--primitive-color-teal-*` |

**Regla**: toda nueva categoría registra su primitivo de color en `08.0` antes de implementarse.

---

## 7. Color en contextos oscuros

La cabecera, el footer y el hero tienen fondo oscuro. Las reglas son:

- Texto sobre oscuro: `#FFFFFF` con opacidad variable según jerarquía.
  - Texto principal: `rgb(255 255 255 / 1.0)`
  - Texto secundario: `rgb(255 255 255 / 0.8)`
  - Texto muted: `rgb(255 255 255 / 0.5–0.65)`
- Los tokens semánticos estándar **no aplican** sobre fondo oscuro. Se declaran variantes locales dentro del componente.
- Los overlays sobre imágenes usan el token `--primitive-color-brand-900` con opacidad, nunca valores crudos.

---

## 8. Lo que el color no puede hacer

- **No puede sustituir a la jerarquía estructural.** Un título no es importante porque es azul; es importante porque ocupa el nivel N1.
- **No puede ser el único diferenciador de un estado.** Un error debe tener color + icono + texto. El color solo no es suficiente (accesibilidad para daltonismo).
- **No puede introducirse ad hoc.** Todo color en uso debe estar en el registro de `08.0`.
- **No puede usarse para decorar** un elemento que no tiene función diferenciada.

---

## 9. Proceso de adición de color

```
1. ¿El color cumple una función estructural o comunica un estado?
   → No: no debe añadirse

2. ¿Ya existe un token semántico que cubra esta función?
   → Sí: usarlo
   → No: continuar

3. ¿Existe el primitivo necesario en 08.0?
   → No: añadirlo con nombre normalizado

4. ¿El ratio de contraste cumple AA?
   → No: ajustar valor hasta cumplir

5. Añadir semántico en 08.0 y documentar su uso aquí
```

---

## Cierre

El color es el nivel superficial del sistema (ver `03-decision-engine`).

Siempre emerge de la estructura.
Nunca la define.
