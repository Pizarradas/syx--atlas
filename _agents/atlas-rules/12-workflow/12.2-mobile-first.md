# 12.2 — Context-first: Atlas traducido a decisiones por mode (SYX Atlas)

## Naturaleza de este documento

El Principio 5 de Atlas (`05.0`) establece que el sistema se diseña desde el peor escenario posible: pantalla pequeña, atención fragmentada, interacción limitada.

Este documento traduce ese principio abstracto a decisiones concretas y verificables para cada mode SYX.

---

## Principio base

> Context-first no es mobile-first.  
> Es worst-case-first.  
> El mobile es solo el caso restrictivo más común.

---

## 1. Qué significa context-first en la práctica

Antes de cualquier decisión de layout, tipo o componente, responder:

```
1. ¿Funciona esto con 320px de ancho?
2. ¿Funciona con atención fragmentada (usuario no concentrado)?
3. ¿Funciona si el usuario lleva 30 segundos en la página?
4. ¿Funciona con dedos (target size mínimo 44×44px)?
5. ¿Funciona si la imagen no carga?
```

Si hay un "no" en cualquiera de estas preguntas → la solución no es válida.

---

## 2. Context-first por mode

### SKETCH — context-first

SKETCH produce bocetos de validación conceptual. La restricción es:

- **Layout**: siempre diseñar la vista de columna única primero. El grid de desktop se añade después si el boceto lo requiere.
- **Contenido**: si un elemento visual no funciona sin imagen (imagen placeholder vacío) → el boceto ya está mal.
- **Interactivos**: indicar el target size en comentarios. No asumir que el usuario usa mouse.

**Añadir siempre al handoff de SKETCH en contexto Atlas:**
```html
<!-- Context-first check:
     ✓ / ✗ 320px — columna única
     ✓ / ✗ Sin imagen
     ✓ / ✗ Target size 44px en interactivos
     Pendiente para UX: [qué queda por resolver] -->
```

---

### UX — context-first

UX mode decide estructura, estados y comportamiento. Las reglas adicionales en contexto Atlas:

#### Layout

- La estructura HTML debe declararse en **orden de prioridad de contenido**, no de prioridad visual en desktop.
- El N1 (hero principal) va primero en el DOM. Siempre.
- En grids de dos columnas (contenido + sidebar): el sidebar va **después** del contenido principal en el DOM. El reordenado visual para desktop se hace con `order` en CSS, nunca con HTML duplicado.

#### Estados

- El estado vacío y el estado sin imagen son obligatorios en cualquier componente editorial. Los componentes de noticias deben funcionar sin imagen.
- En portadas: el estado de carga no puede bloquear el texto. Los titulares se muestran inmediatamente; las imágenes cargan de forma diferida.

#### Interacción

- **Target size mínimo**: 44×44px para cualquier elemento interactivo (etiquetas, botones, controles de datos de mercado, iconos de navegación).
- **Zonas táctiles**: no apilar dos elementos interactivos con menos de 8px entre ellos.
- **Toggle de navegación mobile**: debe tener `aria-expanded` y JS implementado. No puede ser solo CSS.
- **Ticker de datos**: si usa scroll automático, debe detenerse con `prefers-reduced-motion`. El usuario puede pausarlo.

#### Densidad

Los 4 niveles de densidad de `04.4` se implementan así en contexto mobile:

| Nivel densidad | En mobile | En desktop |
|---|---|---|
| D1 — Mínima | Solo N1. Sin grid. | Hero + 1 columna N2 |
| D2 — Baja | N1 + 2 N2 apilados | Hero + 2–3 N2 en grid |
| D3 — Media | N1 + 3 N2 + lista N3 | Hero + grid completo |
| D4 — Alta | N1 + grid comprimido | Grid completo + sidebar |

**Regla**: la densidad D4 en mobile requiere justificación explícita. El contexto de portada no justifica D4 en mobile por defecto.

### TOKEN — context-first

TOKEN mode define los tokens. Las restricciones context-first afectan a dos categorías:

#### Tipografía fluida obligatoria en N1 y N2

Los titulares de mayor jerarquía usan `clamp()`. No hay excepción.

```css
/* N1 — obligatorio */
font-size: clamp(var(--primitive-type-xl), 4vw, var(--primitive-type-3xl));

/* N2 — obligatorio */
font-size: clamp(var(--primitive-type-md), 2vw, var(--primitive-type-lg));
```

Los tokens resultantes son `--semantic-font-size-display` y `--semantic-font-size-heading-lg`.

N3 y N4 usan valores fijos. La fluidez en alta densidad genera irregularidad visual que perjudica la legibilidad.

#### Spacing proporcional en grid

Los tokens de spacing para layout de portada deben ser proporcionales, no absolutos:

```
✓ column-gap: var(--semantic-space-stack-lg)          (24px)
✓ grid-template-columns: 2fr 1fr                      (proporción 2:1)
✗ grid-template-columns: 800px 400px                  (absoluto, viola CL-01)
✗ column-gap: 24px                                    (sin token)
```

### UI — context-first

UI implementa en SCSS. Las restricciones adicionales en contexto Atlas:

#### Pre-flight adicional (antes del checklist estándar R01–R08)

Antes de escribir el primer selector, verificar:

```
□ ¿El layout usa fr, %, min(), max() o clamp()? No px fijos en grid editorial.
□ ¿Los nth-child con índices concretos se usan para gestión de bordes o separadores?
  Si sí → reemplazar por :last-child, :first-child o lógica grid (CC-03 de Atlas).
□ ¿El orden del DOM refleja la prioridad del contenido (N1 primero)?
□ ¿Los titulares N1 y N2 usan clamp()? Verificar tokens existentes en tokens.json.
□ ¿El sidebar está después del contenido principal en el DOM?
```

#### Patrones de layout permitidos

```scss
// ✓ Grid proporcional — columna única colapsando a dos columnas
.org-grid-editorial {
  display: grid;
  grid-template-columns: 1fr; // mobile: columna única

  @include mq(md) {
    grid-template-columns: 2fr 1fr; // desktop: 2:1
  }
}

// ✓ Reordenado visual con order (no duplicar HTML)
.org-grid-editorial__sidebar {
  order: 2; // mobile: después del contenido

  @include mq(md) {
    order: 1; // desktop: columna derecha si el grid lo permite
  }
}
```

#### Patrones de layout prohibidos en páginas editoriales

```scss
// ✗ px fijos en columnas (viola CL-01)
grid-template-columns: 800px 400px;

// ✗ nth-child para bordes entre cards (viola CC-03)
.mol-card-noticia:nth-child(3n) { border-right: none; }

// ✗ position: absolute en publicidad (viola CP-01)
.mol-publicidad { position: absolute; top: 300px; right: 0; }

// ✗ width fijo en imagen de card
.mol-card-noticia__imagen { width: 200px; }  // usar clamp() o %
```

### AUDIT — context-first

AUDIT verifica context-first con estos checks adicionales cuando el archivo es editorial:

| Check | Buscar | Violación si |
|---|---|---|
| `CL-01` Grid sin px | `grid-template-columns` | Contiene valor en `px` |
| `CC-03` nth-child en layout | `nth-child([número])` | Controla bordes o márgenes de columna |
| `CC-05` Toggle con JS | `cursor: pointer` en toggle | No hay `addEventListener` o `onclick` para ese elemento |
| `CA-05` Target size | Elementos interactivos | `width` o `height` < 44px sin compensación de padding |
| `CP-01` Publicidad en flujo | `.mol-publicidad` | Usa `position: absolute` o `position: fixed` |
| Mobile content order | Primer titular en DOM | No es el elemento N1 de la página |
| Clamp en N1 | `atom-headline--3xl` o `--xl` | No usa `clamp()` o no referencia token semántico |

---

### THEME — context-first

THEME crea variantes visuales. Las restricciones context-first son de color y contraste:

- Los tokens de color del tema deben verificarse en fondo blanco Y en fondo oscuro (`--semantic-color-bg-inverse`).
- El contraste mínimo para texto sobre fondo es **4.5:1** (texto normal) y **3:1** (texto grande, ≥ 24px o 18.67px bold).
- Los colores de estado (`--semantic-color-state-success`, `--semantic-color-state-error`) deben ser legibles también en modo alto contraste del sistema operativo.

---

### MIGRATE — context-first

MIGRATE reemplaza variables legacy. En contexto Atlas:

- Si la variable legacy tiene un valor fijo en `px` para layout → el reemplazo debe ser el token proporcional correspondiente (ver `12.1`).
- Si la variable legacy tiene un valor hex hardcodeado → debe convertirse a OKLCH en el token primitivo antes de mapear.
- Verificar que la variable migrada no se usaba en `grid-template-columns` como valor fijo. Si sí → señalar como `CL-01` violation en el reporte de migración.
